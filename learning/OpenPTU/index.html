OpenPTU: Universal Programmable Timing Unit for PIV

OpenPTU is a low-cost, sub-microsecond precision open-source Programmable Timing Unit (PTU) designed specifically for Particle Image Velocimetry (PIV) systems. It serves as a low cost entry point in a world of proprietary, expensive synchronizers by leveraging continuous phase hardware execution to achieve deterministic, phase-locked timing.

Developed in the Gas Dynamics Lab at the Department of Applied Mechanics, IIT Delhi.

Features

  • Multi-Platform Support: Compatible with Arduino UNO (16MHz AVR) and STM32F103C8T6 “Blue Pill” (72MHz ARM Cortex-M3) at present.
  • Continuous Phase Hardware Architecture: Utilizes direct hardware timers without prescaling (TCNT1 for AVR, DWT CYCCNT for ARM) to achieve nanosecond-tier resolution (62.5ns on Uno, 13.8ns on STM32). Executing pulses with sub-microsecond precision.
  • Phase-Locked Execution: By pre-compiling the macroscopic states and utilizing a remainder phase lock, the system never loses synchronization, even across large temporal gaps.
  • TR-PIV Ready: Successfully tested for Time-Resolved PIV frame-straddling operations with extremely short inter-frame times (~ 1μs).

Control Interface (WebSerial API)

OpenPTU features a plug-and-play graphical user interface built on the WebSerial API (ControlUI.html). No heavy desktop software installation is required.

  • Local Execution: Simply open the OpenPTUControlUI.html file in a WebSerial-compatible browser (e.g., Google Chrome, Microsoft Edge, or firefox).
  • Web Access: You can also control the timing unit directly via the web-based controller interface.

Hardware Architecture & Integration

The OpenPTU acts as the central timing hub, coordinating the high-speed camera and the dual-cavity Nd:YAG laser.

Laser Settings

Configure your dual-cavity Nd:YAG laser with the following settings:

  • Trigger set to external all. Now the laser expects both, a flashlamp trigger and a Q-Switch trigger.
  • Get the exact flashlamp to Q-Switch delay parameters from your laser’s technical documentation (tQSW1t_{QSW1}tQSW1​ and tQSW2t_{QSW2}tQSW2​ in OpenPTU Controller UI). These parameters are critical for laser pulse energy and must be input into the OpenPTU controller interface. Typical values are tQSW∼128μst_{QSW}\sim 128\mu stQSW​∼128μs.

Camera Settings

Configure your camera to operate in external trigger and external sync mode with the following settings:

  • Trigger Mode: External (rising edge)
  • Trigger Polarity: Active High
  • Sync Mode: External Sync
  • Sync Polarity: Active High
  • Frame Rate: Set according to the ΔT\Delta TΔT requirement (e.g., 1000 FPS or higher is recommended whenever ΔT\Delta TΔT allows). This fpsfpsfps also needs to be input into the OpenPTU Control UI and must exactly match the value set in the camera.
  • Exposure Time: 1/framerate (e.g., 1ms for 1000 FPS)
  • Frames per trigger: 2 (Most Important). For every given trigger signal, the camera must capture exactly 2 frames. This is critical for frame-straddling operations in TR-PIV. This is also known as frame-on-demand or Random Trigger mode in some cameras.

STM32F103C8T6 (Blue Pill) Version

For the STM32 variant, all timing signals are mapped to Port A (PA2 through PA7). Important: The STM32 operates at 3.3V logic. You must use a logic level shifter (such as the 74AHCT125 quad buffer) to cleanly step up the 3.3V signals to the 5V TTL logic expected by standard cameras and lasers.

Hardware Safety & Isolation

Caution is strongly advised when interfacing with high-power laser electronics. Users are strongly encouraged to place optical isolators on every trigger and sync pin between the MCU and the equipment.

While the microcontrollers generate a maximum of 5V, and a mistimed TTL signal alone will not physically damage the laser or camera, back-EMF and ground loop spikes originating from the laser’s internal power supplies can travel back along the sync cables and severely damage the microcontroller. Although direct, unisolated connections (e.g., routing laser sync directly to camera sync) have been successfully tested in our lab without hardware failure, implementing optical isolation is the professional standard and guarantees long-term electrical safety for your equipment.

Timing Diagram & Execution Model

Traditional software-delay PTUs drift due to operating system overhead and interrupt latencies. OpenPTU solves this via a Compiler-Executor model.

When armed, the unit parses your PIV parameters (frames per second, delta-T, Q-Switch delays) and compiles an absolute chronological timeline of port states and phase targets. During execution, interrupts are disabled and the hardware timer locks onto exact target phases, allowing exact frame-straddling.

Serial Protocol Reference

OpenPTU is controlled via a comma-separated string protocol over 115200 baud serial.

  • Compile & Arm: <ARM, rep_rate_Hz, fps, deltaT_us, offset_us, t_QSW1_us, t_QSW2_us>
    • Example: <ARM,50.0,1000,50,0,120,128>
    • Returns: ACK_ARM,50.0,1000,50
  • Start Execution: <START>
    • Returns: ACK_START (or ERR_NOT_ARMED)
  • Stop Execution: <STOP>
    • Returns: ACK_STOP

Installation

  1. Clone this repository.
  2. Open OpenPTU_ArduinoUno.ino or OpenPTU_STM32F103C8T6.ino in the Arduino IDE based on your hardware.
  3. Select your target board (Arduino UNO or Generic STM32F103C series).
  4. Compile and Upload.
  5. Launch ControlUI.html or click here for the web-based controller interface to connect and configure your timing sequence.
